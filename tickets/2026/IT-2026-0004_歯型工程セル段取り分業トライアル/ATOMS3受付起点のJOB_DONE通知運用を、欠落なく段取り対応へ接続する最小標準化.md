# 1. タイトル
2週間トライアル改善チケット：ATOMS3受付起点のJOB_DONE通知運用を、欠落なく段取り対応へ接続する最小標準化

# 2. 目的（1文＋期限）
ATOMS3でのID受付→同一明細ID受付でJOB_DONE送信→M5呼び出し通知の現行実装を起点に、2週間トライアル中の記録欠落と判断抜けを抑え、後工程まで追える改善ログを残せる運用を成立させる。  
期限：2週間トライアル終了時点で、暫定標準・記録様式・次段実装要件を確立する。

# 3. 現状（事実）
- 現状実装は、ATOMS3でID受付し、同一明細IDの再受付でJOB_DONEを送信する。
- 現場説明は、受付スキャンを「オペID→設備ID→明細ID」の順に2回実施し、2回目でJOB_DONE発生→M5通知→段取り班受付→段取り作業→M5発信なしのため口頭報告、で開始する。
- M5には呼び出し通知が飛ぶが、M5受付時の送信は未実装である。
- WAIT受付は未実装であり、今回の2週間トライアルでは実施しない。
- ログ欠損率5%以下で2週間回すことが既存の成功条件である。
- 現時点では、段取り班の受付結果・完了・保留理由はシステムログに残らず、口頭報告依存である。

# 4. 問題定義（1行）
JOB_DONE起点の呼び出しは動くが、段取り側の受付/完了がログ化されず、E2Eでの滞留と未対応が見えない。

# 5. 真因（暫定→確定へ：構造・ばらつき）
- 暫定真因①：記録点が前工程に偏っており、後工程の反応有無がログに残らない。
- 暫定真因②：M5送信未実装のため、段取り受付・完了・保留が口頭に依存し、ばらつく。
- 暫定真因③：WAIT未実装のため、待ちの構造化はできず、当面は保留理由の最小記録で代替するしかない。
- 暫定真因④：未対応時の時間基準・エスカレーション・記録方法が未標準化である。
- 制約（ボトルネック候補）仮置き：当面の制約候補は、段取り班の受付確認が見えず、管理者が未対応案件を補足しにくい点。

# 6. 封じ込め（今日〜今週）
- 重点1：JOB_DONE後の現場運用を1枚ルールで固定する。
- 重点2：JOB_DONE案件をPower BIで監視し、口頭報告・保留理由はレビュー台帳で補完する。
- 重点3：未対応案件のエスカレーション条件を暫定設定する。
- 重点4：日次レビューで未対応・誤判定・口頭報告漏れを回収する。
- 重点5：WAIT分類はやらず、保留案件のみ短文理由を残す。
- やらないこと：WAIT機能実装、M5全面改修、STOP粒度拡張、分析深掘り、機能追加の拡大。

# 7. 恒久対策（2週間の最小実装：判定基準1枚化→チェック表→裁量境界）
- 判定基準1枚化
  - 同一明細ID再受付をJOB_DONEとみなす条件
  - M5通知受信後の段取り対応ルール
  - 口頭報告を正式記録に読み替える条件
  - 未対応/保留の定義
  - エスカレーション条件
- チェック表
  - 明細ID
  - 設備ID
  - JOB_DONE時刻
  - 状態（対応済/保留/要確認）
  - 口頭報告者
  - 口頭報告時刻
  - 保留理由
  - 備考
- 裁量境界
  - 段取り班が即判断してよい範囲
  - 段取り班リードへ上げる範囲
  - 管理者確認が必要な範囲
  - 要確認：誤JOB_DONE訂正の承認者
- 次段実装要件
  - M5受付送信追加
  - 受付/着手/完了/保留の状態ログ化
  - 誤操作時の扱い（Undo/訂正イベント等）の要件化

# 8. 標準化・定着（標準作業書更新、教育、監査）
- 標準作業書更新
  - 「オペID→設備ID→明細ID×2→JOB_DONE→M5通知→段取り班対応→口頭報告→日次レビュー」の流れを1枚化する。
- 教育
  - トライアル開始前に、加工オペ・段取り班・管理者へ10〜15分で説明する。
  - 特に、2回目スキャンでJOB_DONE発生、M5送信なし、完了は口頭報告、を統一説明する。
- 監査
  - 日次レビューで、Power BI上のJOB_DONE件数とレビュー台帳件数が一致するか確認する。
  - 未記録案件、未対応案件、誤判定疑いを毎日確認する。
- 定着条件
  - JOB_DONE後の全案件が、対応済/保留/要確認のいずれかで追えること。

# 9. 効果確認（1か月：KPI2つ、測り方）
- KPI1：NG件数
  - 定義：JOB_DONE後に未対応・対応不明・誤判定で問題化した件数
  - 測り方：日次レビューで確認し、週次で集計する
- KPI2：記録欠落率
  - 定義：JOB_DONE発生件数に対し、状態がPower BIまたはレビュー台帳で追えない件数の割合
  - 測り方：
    - 分母：ATOMS3から出たJOB_DONE件数
    - 分子：状態、口頭報告、保留理由のいずれかが追えない件数
  - 要確認：既存の欠損率5%定義との整合

# 10. アクション（担当/期限/完了定義）

## A1：トライアル暫定運用ルール1枚化
- 担当：中島翼 / 段取り班リード
- 期限：トライアル開始日まで
- 実施内容：
  - 現場説明を以下で統一する  
    - 受付スキャンは「オペID→設備ID→明細ID」の順に実施  
    - 同一明細IDを2回読むとJOB_DONE発生  
    - M5へ通知  
    - 段取り班が受付し、段取り作業実施  
    - M5発信なしのため、完了は口頭報告
  - JOB_DONE後の扱い、未対応時のエスカレーション、記録方法を1枚にまとめる→記録シート配布　記入担当　青柳
- 完了定義：
  - JOB_DONE後の扱い、未対応時のエスカレーション、記録方法が1枚で共有済み
  - 加工オペ、段取り班、管理者で同じ説明ができる

## A2：JOB_DONE追跡台帳の作成と運用開始→今回やらない
- 担当：段取り班リード（暫定）／要確認：管理者とどちらを主担当にするか
- 期限：今週中
- 実施内容：
  - Power BIでJOB_DONEログを監視する
  - ただしPower BI単体では完了・保留が取れないため、レビュー台帳で補完する
  - 最低限、明細ID・設備ID・JOB_DONE時刻・状態・口頭報告時刻・保留理由を記録する
- 完了定義：
  - 全JOB_DONE案件について、明細ID・時刻・状態がPower BIまたはレビュー台帳で追える
  - 口頭報告漏れ案件が日次レビューで抽出できる

## A3：日次レビュー運用の固定→今回やらない
- 担当：要確認：レビュー司会者
- 期限：今週中
- 実施内容：
  - 日次レビューひな形を作成し、2週間分運用する
  - 毎日確認する項目は以下とする
    - JOB_DONE件数
    - 対応済件数
    - 保留件数
    - 要確認件数
    - 記録欠落件数
    - 重点案件3件まで
  - 記録先、実施時刻、参加者を固定する
- 完了定義：
  - 毎日の実施時刻、参加者、記録先が確定している
  - 2週間継続できるひな形が共有済み

## A4：M5受付送信の改善要件化→今回やらない
- 担当：中島
- 期限：2週間トライアル終了までに要件確定
- 実施内容：
  - M5受付送信の追加要件をチケット化する
  - 最低限の送信イベント項目を整理する
    - terminalId
    - equipmentId
    - detailId
    - eventType
    - eventTs
    - workerId
    - status
  - 状態候補は、CALL_RECEIVED / SETUP_STARTED / SETUP_DONE / HOLD とする
  - 要確認：誤操作時のUndo、訂正イベント、長押し防止
- 完了定義：
  - 送信イベント項目、完了条件、誤操作時の扱いがチケット化済み
  - 次段実装にそのまま渡せる
